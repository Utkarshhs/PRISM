# Insights

All insight types are computed by the backend and returned via `GET /api/dashboard/:productId`. Frontend only renders.

---

## Product Health Score

```
health_score = Σ (feature_positive_rate × feature_weight) × 100
```

Feature weights (must sum to 1.0):
```
performance:       0.20
battery_life:      0.18
build_quality:     0.15
value_for_money:   0.15
customer_support:  0.12
delivery_speed:    0.10
packaging:         0.10
```

`feature_positive_rate` is derived from embedding-based feature extraction (Stage 3). Because feature assignment uses semantic similarity rather than keyword matching, reviews that describe a feature without using its exact name are correctly included in the rate calculation. This makes health scores more accurate and less sensitive to vocabulary variation across platforms or languages.

Range: 0–100. Displayed as a score badge and in the leaderboard across all 4 products.

---

## Product Summary

Steam-style string generated by backend:

> "Moderately Positive (64% positive, 320 reviews), driven by Battery Life (42%) and Packaging (25%) issues"

Sentiment label thresholds:
- ≥80% positive → "Overwhelmingly Positive"
- 65–79% → "Mostly Positive"
- 50–64% → "Moderately Positive"
- 35–49% → "Mixed"
- <35% → "Mostly Negative"

## Feature-Level Sentiment

Per feature: `{ positive_pct, negative_pct, review_count }`. Returned in dashboard response. Rendered as horizontal sentiment bars in the frontend.

Feature assignment is embedding-based: a review is counted toward a feature if its cosine similarity to that feature's anchor phrase exceeds the detection threshold (default 0.50). A single review can contribute to multiple features. Reviews with no feature match above threshold are counted in overall health but not in per-feature bars.

Interactive: clicking a feature tag in the UI filters all panels (graph, trend charts, review list) to that feature.

---

## "What Changed This Week"

Computed in `timeseries.js`. Compares current week vs prior week avg sentiment per feature.
Returns `"up" | "down" | "stable"` per feature. Rendered as a strip of directional arrows (↑↓→).

All signals derive from embedding-based feature extraction, so week-over-week comparisons reflect semantic trends rather than keyword frequency shifts.

---

## Platform Comparison

Same product, side-by-side across platforms. Backend computes health score and feature sentiment per platform. Toggled via a switch in the frontend — no separate API call, data already in dashboard response under `platform_comparison`.

---

## Real-Time Alerts

Threshold: if any feature's negative sentiment rate for the current week exceeds **60%**, alert fires.

Alert is:
1. Stored in `alerts` table in DB
2. Pushed to connected React clients via Socket.io event `"alert:new"`
3. Rendered as a notification badge on the relevant feature panel

## Graph Visualization — Semantic Similarity Network

The graph rendered by the D3 force-directed layout is a **semantic similarity network**, not a keyword-overlap graph. Nodes represent reviews. Edges represent semantic relatedness — driven by embedding-based feature overlap, sentiment alignment, and temporal proximity.

**What this means for the visualization:**

- Nodes that cluster together share the same underlying issue (same feature, same sentiment polarity), even if the reviews use different words
- Cluster density and span directly inform issue classification (systemic vs batch vs isolated)
- Edge thickness is proportional to edge weight (0.0–1.0)
- Node color encodes sentiment: green = positive, red = negative
- Node hover shows review snippet + matched feature + similarity score

Cluster labels (systemic / batch / isolated) are determined by threshold-based connected components — no unsupervised clustering algorithm. This makes classification fully interpretable: a cluster is "systemic" because it has ≥8 nodes spanning ≥3 weeks, not because an algorithm decided so.

## Recommendations Panel

Top N issues (ranked by `confidence × severity_weight`), each rendered as a card:

| Field | Source |
|-------|--------|
| Feature | `issue.feature` |
| Severity | `issue.severity` |
| % impact | `issue.affected_pct` |
| Issue type | `issue.type` (systemic/batch/isolated) |
| Suggested action | Hardcoded action map per feature+type combination in `confidence.js` |

Confidence scores incorporate ambiguity penalties from Stage 3 (sarcasm, mixed polarity, vagueness) and an optional embedding coherence bonus for semantically tight clusters.

**Confidence level display:**

| Score | Color | Label |
|-------|-------|-------|
| 0.75–1.0 | Green | Act immediately |
| 0.45–0.74 | Yellow | Monitor |
| 0.0–0.44 | Red | Gather more data |

## Demo Center — Explainability Output

When `POST /api/demo/run` streams Stage 3 output, it includes per-feature similarity scores so the user can see exactly why a feature was tagged:

```json
{
  "stage": 3,
  "name": "Feature Extraction",
  "method": "pre-trained sentence embeddings (all-MiniLM-L6-v2)",
  "features": {
    "battery_life": {
      "similarity_score": 0.73,
      "anchor": "battery life, charging speed, how long the battery lasts",
      "sentiment": "negative",
      "score": 0.73,
      "ambiguity": null
    },
    "packaging": {
      "similarity_score": 0.61,
      "anchor": "box, packaging, unboxing, damaged box, packaging quality",
      "sentiment": "negative",
      "score": 0.61,
      "ambiguity": "mixed"
    }
  }
}
```

This output is shown in the Demo Center stage card, making the system's reasoning transparent to a non-technical audience.

## PDF Report

Generated server-side via Puppeteer (`server/utils/pdf.js`). Structure:
1. Cover page — product name, date, platform scope
2. Executive summary — health score, top 3 issues
3. Feature-level analysis — sentiment bars per feature
4. Product-level summary — Steam-style summary string
5. Issue deep-dives — type, confidence, affected %, timeline
6. Recommendations — full priority list with actions

Triggered by `POST /api/reports/generate`. Served as static file.
